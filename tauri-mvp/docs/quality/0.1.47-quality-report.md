## AI lifecycle regressions

- Legacy global configuration migrates losslessly to one default profile; a clean install remains profile-empty.
- Two profile-specific local keys remain isolated and cannot overwrite each other.
- Default-profile deletion and Collection Agent references require a replacement before mutation.
- Invalid local-key deletion is rejected without partially deleting or replacing the profile.
- Article task runs snapshot article text, selection, parameters, attachments, and exact profile IDs at creation.
- Faster successful results become readable while other profiles remain pending; a local error does not block another profile.
- Cancellation stops local adoption, avoids blindly starting queued provider futures, and permits a later run.
- Reconnection calls only the status endpoint and never creates a duplicate run.
- UTF-16 browser offsets convert to Python code-point offsets before write-back, including text containing emoji.
- Apply is idempotent, blocks article drift, and creates an `AI_BEFORE_APPLY` version before changing text.
- A late article-chat reply or failure cannot pollute the article selected afterward.

A real DeepSeek + GLM article task used a disposable synthetic article. Both profiles succeeded through Chat Completions; DeepSeek completed in 25.37s and GLM in 5.93s. The task was cleared, the disposable article was deleted, and article count remained unchanged. The request contained exactly those two profile IDs.

The earlier GLM credential produced a deterministic sanitized 403 account-group denial. After the user explicitly supplied a replacement key, the same GLM profile and model passed without changing the profile's provider, model, base URL, or transport. This confirms the earlier failure was credential/account access rather than hidden fallback or duplicate request behavior.

No profile was silently switched to another credential or model to manufacture a passing result. Timeout, 429, and temporary-network errors are not automatically retried or deleted.

## Error and privacy audit

- Screen-level errors use the shared safe formatter rather than exposing raw exception messages.
- Missing key/login, login timeout, 401, 403, 404, 429, timeout, HTML, and traceback-shaped responses map to actionable public diagnostics.
- Health records store status, time, transport, latency, and safe diagnostic code/message only; they do not store model reply text or a secret.
- The final source scan found no key-shaped secret, private writing, private screenshot, database, log, or test-result artifact in the planned commit.
- The unpacked application and sidecar plus both final installer files contain no key-shaped secret, database signature, test artifact, temporary path, repository path, or local Windows username under the release scan patterns.
- Release Rust builds remap the local Cargo and repository prefixes to generic build paths so panic metadata does not publish the build machine username.
- `git diff --check` and the screen-level unsafe-error audit passed. The CI exclusion for the shared safe formatter now uses a repository-root-safe glob.
