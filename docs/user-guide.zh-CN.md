# 活着为了讲述官方手册

活着为了讲述（Living to Tell）是一个本地优先的桌面写作工作室，用来写文章、编排作品集、管理文脉和意象，并在你主动选择时调用 AI。这个手册按任务组织：先告诉你适合做什么，再给出具体步骤、注意事项和常见问题。

快速演示见 [GIF 教程](tutorials.zh-CN.md)。

## 1. 安装、数据与安全边界

适合做什么：确认程序文件和写作数据分开，知道卸载、升级、备份分别会影响什么。

步骤：

1. 从 [GitHub Releases](https://github.com/sidiangongyuan/living-to-tell/releases/tag/living-to-tell-v0.1.32) 下载最新 Windows 预览版。
2. 推荐运行 `LivingToTell_0.1.32_x64-setup.exe`；也可以使用 `LivingToTell_0.1.32_x64_zh-CN.msi`。
3. 从开始菜单或桌面快捷方式打开 **活着为了讲述**。
4. 进入 **设置 → 数据与存储**，查看当前 SQLite 数据库、备份目录和检查点目录。
5. 大篇幅修改前，进入 **导出与备份** 创建一次备份或命名检查点。

注意事项：

- 写作数据库默认保存在 `%APPDATA%\LivingToTell\LivingToTell\living-to-tell.sqlite3`。
- 程序通常安装在 `%LOCALAPPDATA%\活着为了讲述`，这和写作数据库不是同一个位置。
- 卸载应用不会删除文章数据库、备份或检查点。
- 数据目录迁移采用复制方式，旧目录会完整保留。
- 如果存在旧 Writer 数据库 `%APPDATA%\Writer\Writer\writer.sqlite3`，首次启动会复制到新目录；旧数据库不删除。

常见问题：

- **Windows 提示未签名风险**：当前预览版暂未签名，只运行本仓库 Release 页下载的安装包。
- **升级后数据会不会没了**：正常升级不会删除写作数据库。升级前仍建议创建检查点。
- **要不要手动复制数据库**：可以，但更推荐用应用内备份/检查点，路径和恢复入口更清楚。

## 2. 首次使用与示例项目

适合做什么：第一次打开时快速理解完整工作流，而不污染自己的真实作品库。

步骤：

1. 打开 **日期** 页。
2. 查看欢迎清单：第一篇文章、第一条文脉、AI 设置、备份位置、文章对话。
3. 如果想快速体验完整流程，点击 **创建示例项目**。
4. 点击 **打开作品集**，查看示例文章、作品集大纲、文脉标本、文章便签和场景 AI Card。
5. 体验结束后，回到日期页点击 **删除示例**。

注意事项：

- 示例项目不会自动创建，必须手动点击。
- 删除示例项目只删除应用记录在 marker 中的精确 ID，不按标题或标签模糊删除。
- 你自己创建的同名文章、同名标签、同名作品集不会因为删除示例而被清理。

常见问题：

- **欢迎清单不见了**：进入 **设置**，使用“重新显示欢迎清单”。
- **示例项目是否能长期保留**：可以，它只是普通本地内容；不需要时再删除。

## 3. 文章写作

适合做什么：写随笔、短篇、场景、人物笔记、观察记录和长文草稿。

步骤：

1. 从 **日期** 页点击空日期的 **开始写作**，或从左侧导航进入 **文章**。
2. 在左侧列表选择文章，右侧是正文编辑区。
3. 直接输入正文；应用会自动保存。
4. 给文章添加标签，方便以后搜索和筛选。
5. 在右侧上下文栏写文章便签，用来记录临时提醒、下一步写法或删掉的片段。
6. 打开 **历史版本**，在大改前点击 **保存当前版本**。
7. 需要沉浸写作时进入专注模式，只保留正文区域。

注意事项：

- 文章便签不进入正文、导出和全文搜索。
- 历史版本只保存关键节点：手动保存、AI 写回前、恢复前。
- AI 结果不会自动写回正文，必须你明确选择替换、插入或复制。

常见问题：

- **正文开头缩进会不会丢**：当前保存逻辑会保留正文开头的全角空格。
- **切换文章偶尔显示 Not Found**：旧请求会被忽略；如果当前文章真的不存在，应用会刷新列表并显示中文提示。
- **长文末尾写作不舒服**：编辑器保留文末空白，方便在底部继续写。

## 4. 作品集与长篇大纲

适合做什么：把多篇文章编成一个阅读顺序，或把长篇项目拆成分部、章节、场景和笔记。

步骤：

1. 进入 **作品集**。
2. 点击 **新建作品集**，填写标题和说明。
3. 点击加入文章，把已有文章收进作品集。
4. 在稿件视图中调整文章顺序，并用右侧预览检查阅读节奏。
5. 切到 **大纲**，新增分部、章节、场景或笔记。
6. 给大纲项设置状态、摘要、视角、时间线、场景地点、标签和目标字数。
7. 切到 **规划看板**，按构思、草稿、修订、完成、暂停查看全局状态。
8. 需要时从大纲项一键创建文章，或把现有文章关联到大纲项。
9. 导出作品集为 Markdown、TXT 或 DOCX。

注意事项：

- 作品集只保存文章关系和顺序，不复制正文。
- 从作品集中移除文章不会删除原文章。
- 大纲不改数据库结构，使用现有作品集大纲字段保存。

常见问题：

- **作品集适合长篇小说吗**：当前适合 V1 长篇规划：结构、状态、目标字数、关联文章。更复杂的出版格式以后再做。
- **可以只写大纲不写正文吗**：可以，大纲项可以先不关联文章。

## 5. 文脉库与意象星图

适合做什么：保存以后会反复参考的摘录、风格样本、意象、技法、人物材料和来源信息。

步骤：

1. 进入 **文脉库**。
2. 新建文脉标本，填写正文、书名/篇名、作者、用途和个人笔记。
3. 在日期页的每日文脉卡片中，可以跳回对应标本。
4. 在文章或文脉正文中选中文字后右键，选择 **加入意象星图**。
5. 在弹窗中搜索已有意象或创建新意象，保存摘录。
6. 进入 **意象星图**，查看节点大小、颜色和连线关系。
7. 在意象详情点击 **AI 丰富**，可把“神话模式”“奴隶道德”“常人”等概念生成短卡；预览后再选择追加或覆盖笔记。
8. 在意象详情中移除摘录时，只会解除当前意象链接；同一摘录在其他意象中仍保留。

注意事项：

- 文脉库保存来源和用途；意象星图保存“这句话属于哪些反复出现的意象”。
- 同一来源同一选区会复用同一摘录，不重复创建。
- 如果文章修改导致位置漂移，应用会尝试用文本和上下文找回原摘录并合并重复项。
- AI 丰富不会自动创建关联节点或语义边；关联建议只是提示，星图连线仍来自真实摘录共现。

常见问题：

- **同一句话能加入多个意象吗**：可以，一条摘录可以同时连接多个意象。
- **删除一个意象里的摘录会不会删掉其他意象里的同一句话**：不会，默认只是从当前意象移除。

## 6. AI、AI 卡片与多模型对比

适合做什么：润色、改写、扩写、续写、总结、拟标题、结构诊断，以及把风格/人物/场景卡作为可复用上下文。

步骤：

1. 进入 **设置 → AI**，选择 OpenAI 兼容接口、Gemini、Gemini CLI 或 OpenCode。
2. 先点 **检查本地配置**，确认凭据来源存在。
3. 再点 **发送真实测试请求**，用短请求验证 provider、model、base URL、key 和 transport 真的可用。
4. 如果要多模型对比，在 **AI 配置档案** 中导入或创建 GPT / Gemini / DeepSeek / OpenCode 等档案。
5. 进入 **AI 工作区**，选择任务、范围和输入文本。
6. 勾选最多 3 个模型档案，运行同一任务。
7. 比较每张结果卡的字数、段落、耗时、token 和成本。
8. 选择胜出结果后，再复制、替换或插入正文。
9. 进入 **AI Cards**，创建风格、人物或场景卡；也可以用卡片生成 AI 先生成草稿，再手动保存。

注意事项：

- AI 提供方只会收到你主动发送的文本。
- “检查本地配置”不是远端可用证明；只有“发送真实测试请求”成功才代表模型可用。
- 场景模块需要在 AI 工具中手动搜索并勾选；未勾选的卡片不会进入上下文。
- AI 卡片适合保存结构，不适合保存大段原文。

常见问题：

- **OpenCode 是否需要在应用里保存 key**：不需要，应用调用本机 `opencode auth login` 后的 CLI 登录状态。
- **Gemini 中转 key 为什么以前 403**：如果是 `sk-...` 中转 key + 自定义 Base URL，应用会自动走兼容的 chat completions transport。
- **AI 结果会不会自动覆盖文章**：不会，写回必须你明确确认。

## 7. 导出与备份中心

适合做什么：在大改前留恢复点，把文章或作品集导出成外部文件，并确认数据路径。

步骤：

1. 进入 **导出与备份**。
2. 查看安全摘要：最近恢复点距离现在多久、备份数量、检查点数量、总大小。
3. 大改前点击 **创建检查点**，写一个能看懂的名字和说明。
4. 也可以点击 **创建备份**，生成自动备份。
5. 在恢复点列表中选择一个备份或检查点。
6. 如果需要恢复，点击 **恢复所选**；应用会在恢复前自动备份当前数据库。
7. 在 **数据位置** 中复制或打开数据库、备份、检查点目录。
8. 在导出快捷操作中导出最近文章或最近作品集。

注意事项：

- 恢复入口只使用应用管理的备份或检查点。
- 恢复前会自动备份当前数据库，降低误恢复风险。
- 路径显示用于确认和复制，不会修改数据。

常见问题：

- **多久备份一次**：高频写作建议每天或每 3 天；普通使用每 7 天也可以。
- **导出和备份有什么区别**：导出是给人读的 TXT/Markdown/DOCX；备份是给应用恢复数据库用的 SQLite 副本。

## 8. 公开截图、反馈与隐私检查

发布截图、问题反馈、日志或示例文件前，请检查：

- 使用演示文本，不使用私人写作内容。
- 不包含真实 API key、OAuth token、账号邮箱或云项目 ID。
- 不包含本地数据库文件。
- 不暴露真实本机路径；如需说明路径，使用通用示例。
- 报 bug 时尽量描述：页面、操作顺序、看到的错误文案、是否刚切换文章或刚升级版本。
